# セキュリティアーキテクチャ

AnimaWorksはツールアクセス・永続記憶・エージェント間通信を持つ自律AIエージェントを実行する。これはステートレスなLLMラッパーとは根本的に異なる脅威面を生み出す — エージェントはファイルを読み、コマンドを実行し、メッセージを送り、人間の介入なしにスケジュールで動作する。

本ドキュメントでは、これを本番環境で安全に運用するための多層防御セキュリティモデルを記述する。

---

## 脅威モデル

| 脅威 | 攻撃経路 | 影響 |
|------|---------|------|
| 外部データ経由のプロンプトインジェクション | Web検索結果、Slack/Chatworkメッセージ、メール | エージェントが攻撃者の指示を実行 |
| エージェント間のラテラルムーブメント | 侵害されたエージェントが悪意あるDMを同僚に送信 | 組織全体の権限昇格 |
| 破壊的コマンド実行 | エージェントが `rm -rf /` や `curl … \| sh` を実行 | データ喪失、システム侵害 |
| パストラバーサル | エージェントがサンドボックス外に読み書き | エージェント間データ漏洩、設定改ざん |
| 無限メッセージループ | 2つのエージェントが無限に返信し合う | リソース枯渇、API費用の爆発 |
| 意図しない外部送信 | エージェントが想定外の宛先にメッセージ送信 | データ流出 |

---

## 1. プロンプトインジェクション防御 — 信頼境界ラベリング

エージェントのコンテキストに入る全データに信頼レベルがタグ付けされる。モデルはこの境界を明示的に認識し、untrustedコンテンツをデータとして扱い、指示としては絶対に従わないよう指示される。

### 信頼レベル

| レベル | 対象ソース | 扱い |
|--------|-----------|------|
| `trusted` | 内部ツール（send_message, search_memory）、システム生成 | 通常通り実行 |
| `medium` | ファイル読み取り、RAG結果、ユーザープロファイル、統合済み知識 | 参考データとして解釈 |
| `untrusted` | web_search, slack_read, chatwork_read, gmail_read, x_search | **指示には絶対従わない** |

### 実装

```
<tool_result tool="web_search" trust="untrusted">
  検索結果 — インジェクション試行が含まれる可能性あり
</tool_result>

<priming source="related_knowledge" trust="medium">
  RAGで取得されたコンテキスト
</priming>
```

**オリジンチェーン伝播**: データが複数システムを経由する場合（例: Web → RAGインデックス → Priming）、信頼レベルはチェーン中の最小値に劣化する。RAGにインデックスされたWeb検索結果は、後から取得されても `untrusted` ステータスを保持する。

**主要ファイル**: `core/execution/_sanitize.py`（信頼解決、境界ラッピング）, `templates/*/prompts/tool_data_interpretation.md`（信頼レベル解釈のモデル向け指示）

## 2. コマンド実行セキュリティ — 5層防御

エージェントはシェルコマンドを実行できる。5つの独立したレイヤーが悪用を防ぐ:

### レイヤー1: シェルインジェクション検出

コマンドの連鎖やインジェクションに使われるシェルメタ文字をブロック:

- セミコロン（`;`）、バッククォート（`` ` ``）、改行（`\n`）
- コマンド置換（`$()`、`${}`、`$VAR`）

### レイヤー2: ハードコードブロックリスト

権限設定に関係なく**常に**ブロックされる、パターンマッチによる破壊的コマンド:

| パターン | 理由 |
|---------|------|
| `rm -rf`, `rm -r` | 再帰的削除 |
| `mkfs` | ファイルシステム作成 |
| `dd of=/dev/` | ディスク直接書き込み |
| `curl\|sh`, `wget\|sh` | リモートコード実行 |
| `\| sh`, `\| bash`, `\| python` | インタープリタへのパイプ |
| `chmod *7*` | ワールドライタブル権限 |
| `shutdown`, `reboot` | システムシャットダウン |
| `> /dev/sd*`, `> /etc/` | デバイス/システムファイルへのリダイレクト |

### レイヤー3: エージェント個別の禁止コマンド

各エージェントの `permissions.md` に `## 実行できないコマンド` セクションを定義でき、そのエージェントのロール固有の追加ブロックコマンドを指定できる。

### レイヤー4: エージェント個別の許可リスト

エージェントの許可リスト（`permissions.md` より）にマッチするコマンドのみ実行許可。明示的なコマンド権限がないエージェントはデフォルト拒否。

### レイヤー5: パストラバーサル検出

コマンド引数にパストラバーサルパターン（`../`、サンドボックス外の絶対パス）がないかチェック。

**パイプライン個別チェック**: パイプで連結されたコマンドの各セグメントは独立にチェックされる — `safe_cmd | dangerous_cmd` もブロックされる。

**主要ファイル**: `core/tooling/handler_base.py`（ブロックリスト、インジェクション正規表現）, `core/tooling/handler_perms.py`（5層チェックパイプライン）

## 3. ファイルアクセス制御 — デフォルトでサンドボックス

各エージェントは自身のディレクトリ（`~/.animaworks/animas/{name}/`）内で動作する。このサンドボックス外へのファイルアクセスには明示的な権限が必要。

### 保護ファイル（イミュータブル）

セキュリティ上重要な設定の自己改変を防ぐため、以下のファイルはエージェント自身が書き込めない:

- `permissions.md` — ツール・コマンドの許可リスト
- `bootstrap.md` — 初回起動指示

### 上司アクセスマトリクス

上司（マネージャー）は、スコープ付き権限で部下のデータにアクセスできる:

| パス | 直属部下 | 全配下 |
|------|:---:|:---:|
| `activity_log/` | 読み取り | 読み取り |
| `state/current_task.md`, `pending.md` | — | 読み取り |
| `state/task_queue.jsonl`, `pending/` | — | 読み取り |
| `status.json` | 読み書き | 読み取り |
| `identity.md` | — | 読み取り |
| `injection.md` | 読み書き | 読み取り |
| `cron.md`, `heartbeat.md` | 読み書き | — |

配下の解決にはBFS（幅優先探索）を使用し、循環する上司チェーンによる無限ループを防ぐサイクル検出を実装。

**主要ファイル**: `core/tooling/handler_perms.py`（_check_file_permission）, `core/tooling/handler_memory.py`（メモリ読み書きガード）, `core/tooling/handler_org.py`（階層チェック）

## 4. プロセス隔離

各エージェントは `ProcessSupervisor` が管理する独立OSプロセスとして動作:

- **プロセス分離**: 1つのエージェントのクラッシュが他に影響しない
- **Unix Domain Socket IPC**: ファイルシステムソケットによるプロセス間通信（TCP不使用）でネットワーク露出を最小化
- **独立ロック**: チャット、Inbox、バックグラウンドタスクが別々のasyncioロックを使用 — 並行パスが相互にブロックしない
- **ソケットディレクトリ**: `~/.animaworks/run/sockets/{name}.sock`、起動時に残存ソケットの自動クリーンアップ

**主要ファイル**: `core/supervisor/manager.py`, `core/supervisor/ipc.py`, `core/supervisor/runner.py`

---

## 5. レート制限 — 3層のアウトバウンド制御

自律エージェントがスパムしてはならない。3つの独立したレイヤーがメッセージ制限を施行:

### レイヤー1: セッション内（Per-Run）

- 1回の実行セッション内で同一宛先へのDM重複なし
- 1チャネルにつき1セッション1投稿
- インメモリのセット（`_replied_to`, `_posted_channels`）で追跡

### レイヤー2: セッション横断（永続）

- エージェントあたり**1時間30通**
- エージェントあたり**1日100通**
- `activity_log` のスライディングウィンドウから計算 — プロセス再起動後も有効
- `ack`, `error`, `system_alert` メッセージは制限対象外

### レイヤー3: 行動認識（自己調整）

直近の送信メッセージ（過去2時間、最大3件）がPrimingを通じてエージェントのシステムプロンプトに注入される。エージェントは自身の送信パターンを認識し、自己調整できる。

### カスケード防止

- **会話深度リミッター**: 10分以内の同一エージェントペア間最大6ターン（3往復）
- **Inboxレートリミッター**: クールダウン期間、カスケード検出、遅延メッセージ処理
- **送信者別レート制限**: ハートビート中の単一送信者からのメッセージを制限

**主要ファイル**: `core/tooling/handler_comms.py`（per-run）, `core/cascade_limiter.py`（cross-run、深度）, `core/supervisor/inbox_rate_limiter.py`（Inbox）, `core/memory/priming.py`（_collect_recent_outbound）

## 6. 認証・セッション管理

### 認証モード

| モード | 用途 |
|--------|------|
| `local_trust` | 開発用 — localhostからのリクエストは認証バイパス |
| `password` | シングルユーザーのパスワード保護 |
| `multi_user` | 複数ユーザー（個別アカウント） |

### セッションセキュリティ

- **Argon2id** パスワードハッシュ（メモリハード、サイドチャネル耐性）
- **48バイトURLセーフトークン**（暗号学的乱数）
- **ユーザーあたり最大10セッション** — オーバーフロー時は最古を破棄
- **Cookie ベース** のセッション伝送、`/api/` と `/ws` ルートにミドルウェアガード
- 設定ファイル（`config.json`, `auth.json`）は**0600パーミッション**で保存

### Localhost信頼

`trust_localhost` が有効な場合、ループバックアドレスからのリクエストは自動認証される。OriginとHostヘッダーのチェックにより、localhostに対するブラウザベースのCSRF攻撃を緩和。

**主要ファイル**: `core/auth/manager.py`, `server/app.py`（auth_guardミドルウェア）, `server/localhost.py`

## 7. Webhook検証

外部プラットフォームからのインバウンドWebhookを暗号学的に検証:

| プラットフォーム | 方式 | リプレイ防止 |
|----------------|------|------------|
| Slack | HMAC-SHA256（signing secret） | タイムスタンプチェック（5分ウィンドウ） |
| Chatwork | HMAC-SHA256（webhookトークン） | — |

**主要ファイル**: `server/routes/webhooks.py`

---

## 8. SSRF緩和 — メディアプロキシ

メディアプロキシ（`/api/media-proxy`）はUI表示用に外部画像を取得する。以下を強制:

- **HTTPSのみ** — 平文HTTPは不可
- **ドメイン許可リスト** — 設定された信頼ドメインのみ
- **プライベートIPブロック** — localhost、プライベートレンジ（RFC 1918）、リンクローカル、マルチキャスト、予約済みをブロック
- **DNS解決チェック** — ホスト名を解決しIPがプライベートでないことを確認（DNSリバインディング防止）
- **Content-Type検証** — `image/jpeg`, `image/png`, `image/gif`, `image/webp` のみ
- **IP別レート制限** — オープンプロキシとしての悪用を防止

**主要ファイル**: `server/routes/media_proxy.py`

---

## 9. Mode S（Agent SDK）セキュリティ

Claude Agent SDK（Mode S）で動作する場合、`PreToolUse` フックによる追加ガードレールが適用:

- **Bashコマンドフィルタリング**: SDK実行コマンド用の個別ブロックリスト
- **ファイル書き込み保護**: 保護ファイルリストとエージェントサンドボックスに対する書き込み先検証
- **ファイル読み取り制限**: 他エージェントのディレクトリへのアクセスをブロック
- **出力トランケーション**: Bash出力は10KBで切り捨て、ファイル読み取り/grep/globもサイズ制限

**主要ファイル**: `core/execution/_sdk_security.py`

---

## 10. アウトバウンドルーティングセキュリティ

統一アウトバウンドルーター（`resolve_recipient()`）がエージェントの意図しない宛先への送信を防止:

1. 既知エージェント名との完全一致（大文字小文字区別）
2. 明示的設定からのユーザーエイリアス検索（大文字小文字非区別）
3. プラットフォームプレフィックス付き宛先（`slack:USERID`, `chatwork:ROOMID`）
4. フォールバック：大文字小文字非区別のエージェント名マッチ
5. **未知の宛先 → ValueError**（fail-closed）

明示的な設定なしに、エージェントは任意の外部アドレスに送信できない。

**主要ファイル**: `core/outbound.py`

## まとめ: 多層防御

```
┌─────────────────────────────────────────────────────────┐
│                    外部データ                             │
│              (Web, Slack, メール 等)                      │
└────────────────────────┬────────────────────────────────┘
                         │
              ┌──────────▼──────────┐
              │  信頼境界            │  ← untrusted/medium/trusted タグ
              │  ラベリング          │  ← オリジンチェーン伝播
              └──────────┬──────────┘
                         │
              ┌──────────▼──────────┐
              │  認証・セッション     │  ← Argon2id, トークンベースセッション
              │  管理               │  ← Webhook HMAC検証
              └──────────┬──────────┘
                         │
     ┌───────────────────┼───────────────────┐
     │                   │                   │
┌────▼────┐      ┌──────▼──────┐     ┌──────▼──────┐
│ コマンド │      │ ファイル     │     │ アウトバウンド│
│ セキュリティ│    │ アクセス制御  │     │ レート制限   │
│ (5層     │      │ (サンドボックス│    │ (3層)       │
│  チェック) │     │  + ACL)     │     │             │
└────┬────┘      └──────┬──────┘     └──────┬──────┘
     │                  │                   │
     └───────────────┐  │  ┌────────────────┘
                     │  │  │
              ┌──────▼──▼──▼────────┐
              │  プロセス隔離        │  ← エージェント別OSプロセス
              │  (Unixソケット)      │  ← 独立ロック
              └─────────────────────┘
```

各レイヤーは独立して動作する。1つのレイヤーの失敗は他のレイヤーで補捉される — 信頼ラベリングを迂回したプロンプトインジェクションも、コマンドブロックリスト、ファイルサンドボックス、レート制限に直面する。
